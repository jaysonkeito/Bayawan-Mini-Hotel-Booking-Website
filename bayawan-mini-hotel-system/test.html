📋 Executive Summary

This is a well-architected PHP hotel booking system with a clean separation of concerns, proper use of prepared statements throughout, a centralized .env-based config system, and a thoughtful rate-limiting implementation. The codebase is clearly the work of a developer who understands security fundamentals. The two critical issues below are fixable in under an hour, and several medium-priority improvements would bring this to a production-ready standard.

🔒 Security Audit

✅ What's Well Done

Strength
Prepared Statements Used Consistently

All database queries in admin_configuration.php's helper functions (select(), insert(), update(), delete()) use mysqli_prepare() with bound parameters. SQL injection is effectively prevented project-wide.

Strength
Passwords Handled Correctly

password_hash(PASSWORD_DEFAULT) on storage, password_verify() on login. The profile handler even has an explicit comment explaining why raw $_POST must be read for passwords instead of filteration() — avoiding a subtle htmlspecialchars corruption bug on special-character passwords.

Strength
Rate Limiting Implemented (Both Session + DB)

Login, OTP, and registration flows all use a session-based rate limiter with lockout. The contact form uses a separate DB-backed IP rate limiter. 5 attempts → 15-minute lockout is a solid baseline.

Strength
Remember-Me Token is Cryptographically Secure

bin2hex(random_bytes(32)) generates 64-character unpredictable tokens. The cookie is set with HttpOnly and Secure flags.

Strength
.env Excluded from Version Control

.gitignore correctly excludes .env and only a safe .env.example is committed. No secrets are hardcoded anywhere in the codebase.

🚨 Critical Issues

Critical
SQL Injection via Raw ID in last_login UPDATE

In ajax/user_login_register.php, after a successful login the last_login timestamp is updated with a raw session value injected directly into the query string:

// VULNERABLE — $row['id'] is from DB, but this pattern is still unsafe
$conn->query("UPDATE user_cred SET last_login = NOW() WHERE id = {$row['id']}");

Although $row['id'] comes from a database result and is likely an integer, it bypasses the prepared-statement convention used everywhere else. A coding error elsewhere could introduce a tainted value through this pattern.

// FIX — use a prepared statement consistently
$upd = $conn->prepare("UPDATE user_cred SET last_login = NOW() WHERE id = ?");
$upd->bind_param("i", $row['id']);
$upd->execute();
$upd->close();
Critical
Password Reset Reuses the remember_token Column

The forgot-password flow stores the reset token in remember_token — the same column used by the "Remember Me" login feature. This means:

  • Triggering a password reset invalidates all existing "Remember Me" sessions for that user.
  • The recover_password case has no expiry check — a reset link is valid forever until used.
// FIX — add dedicated columns to user_cred:
ALTER TABLE user_cred
  ADD COLUMN reset_token VARCHAR(64) NULL,
  ADD COLUMN reset_expires DATETIME NULL;

// Then check expiry in recover_password:
$stmt = $conn->prepare(
  "SELECT id FROM user_cred
   WHERE email=? AND reset_token=? AND reset_expires > NOW() LIMIT 1"
);

⚠️ Warnings

Warning
SVG Upload Has No Content Sanitization

uploadSVGImage() in admin_essentials.php only checks the MIME type and file size, then calls move_uploaded_file() directly. SVG files can contain embedded JavaScript (<script> tags) and are a well-known XSS vector when served from the same origin.

// FIX — strip script tags and event handlers before saving
$svg_content = file_get_contents($image['tmp_name']);
$svg_content = preg_replace('//i', '', $svg_content);
$svg_content = preg_replace('/\bon\w+\s*=/i', 'data-removed=', $svg_content);
file_put_contents($path, $svg_content);
Warning
CSRF Tokens Not Implemented on AJAX Forms

All AJAX form submissions (login, registration, booking, profile update) rely on session-based authentication but do not include CSRF tokens. A malicious page could trigger cross-site requests if the user is logged in. This is especially important for payment and booking endpoints.

// Minimal CSRF implementation — generate once per session
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
// In each AJAX handler:
if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'] ?? '')) {
    http_response_code(403); exit('CSRF validation failed');
}
Warning
OTP Is Compared with Loose Equality (==)

In ajax/user_login_register.php, the OTP check uses == instead of ===. In PHP, "0" == false and similar type-coercion oddities can cause unexpected matches. Always use strict comparison for security-sensitive checks.

// Vulnerable
if ($otp == $_SESSION['reg_otp']) { ... }

// Fixed
if ($otp === (string)$_SESSION['reg_otp']) { ... }
Warning
get_client_ip() Trusts X-Forwarded-For Without Proxy Validation

The IP detection in rate_limiter.php checks HTTP_X_FORWARDED_FOR first. This header is easily forged by clients unless you're behind a trusted reverse proxy. An attacker can bypass the DB-based rate limiter (contact form) by rotating this header value.

// Only trust X-Forwarded-For if your server IS behind a known proxy.
// Otherwise, use only REMOTE_ADDR:
function get_client_ip(): string {
    return filter_var($_SERVER['REMOTE_ADDR'], FILTER_VALIDATE_IP)
        ? $_SERVER['REMOTE_ADDR']
        : '0.0.0.0';
}
Warning
Payment Records Created Before Successful Payment Confirmation

In user_pay_now.php, booking rows are inserted into booking_order and booking_details before PayMongo confirms payment. The cart flow does mark these as "payment failed" if checkout creation fails, but if the user simply abandons the PayMongo page, orphaned "pending" records accumulate. A cleanup cron job or scheduled query should purge stale pending bookings older than 30–60 minutes.

🐛 Bug Fixes

Bug
filteration() Applied to json_decode() Output in admin_rooms.php

On add_room and edit_room, json_decode($_POST['features']) returns an array of integers, but then filteration() is called on it. filteration() expects an associative array of strings and calls htmlspecialchars() on each value — converting integer IDs to HTML-encoded strings. These are then passed as bound parameters. While MySQLi will cast them back, the function misuse is fragile and could silently corrupt non-integer decoded values in the future.

// FIX — cast decoded JSON values to int directly, skip filteration()
$features   = array_map('intval', (array)json_decode($_POST['features']   ?? '[]'));
$facilities = array_map('intval', (array)json_decode($_POST['facilities'] ?? '[]'));
Bug
Booking Availability Check Uses session_start() Inside a Conditional

In ajax/user_confirm_booking.php, session_start() is called inside the if(isset($_POST['check_availability'])) block — meaning it only starts when that specific POST key exists. If other functionality is ever added to this file, sessions will be unavailable. Move session_start() to the top of the file unconditionally.

Bug
DOB Validation in complete_google_profile Is Too Loose

The check if (!$dob_ts || $dob_ts >= strtotime('-1 year')) only rejects dates less than 1 year ago. A user could enter a date 2 years ago or even a future date that is over 1 year ahead. A more complete validation should reject future dates and require a minimum age (e.g., 18 years).

$dob_ts  = strtotime($dob);
$min_age = strtotime('-18 years');
if (!$dob_ts || $dob_ts > time() || $dob_ts > $min_age) {
    echo 'Please enter a valid date of birth (must be 18+).';
    exit;
}
Bug
uploadImage() Trusts MIME Type from $_FILES, Not File Content

The $_FILES['type'] value is provided by the browser and can be spoofed. A user could upload a PHP file disguised as a JPEG. Use finfo_file() to check the actual file magic bytes.

$finfo = new finfo(FILEINFO_MIME_TYPE);
$actual_mime = $finfo->file($image['tmp_name']);
$valid_mime  = ['image/jpeg', 'image/png', 'image/webp'];
if (!in_array($actual_mime, $valid_mime)) return 'inv_img';

Architecture Overview

The system follows a procedural PHP + AJAX pattern with two distinct authenticated contexts: User (guest-facing) and Admin (hotel staff). All pages share a central config loaded via config/env.php. Database operations are wrapped in four typed helpers (select, insert, update, delete) defined in admin_configuration.php.

Booking State Machine

Bookings move through the following statuses in booking_order.booking_status:

pending  →  booked  →  checked_in  →  checked_out
   ↓              ↓
payment       cancelled
 failed

Payment confirmation is handled in user_pay_response.php, which reads the PayMongo session ID from $_SESSION and calls the PayMongo API to verify payment before updating status to booked.

Rate Limiting Configuration

Two constants in includes/rate_limiter.php control all rate limiting:

RATE_MAX_ATTEMPTS    = 5          // attempts before lockout
RATE_LOCKOUT_SECONDS = 900        // 15 minutes

Session-based limits (session_rate_limit) apply per browser session and are used for login, OTP, and registration. DB-based limits (db_rate_limit) apply per IP address and persist across sessions — used for the contact form.
